# План (упрощённый): сначала только нейросеть, без скриптов

## Зачем упрощение

Нужно **показать применимость нейросети** в вашей работе по НК. Отдельно **скрипты не разрабатываем в этой фазе** — с ними и так всё понятно; время уходит на промпты, оркестрацию и **простой текстовый отчёт** (Markdown), без возник с форматом Excel.

Полный вариант плана (скриптовые пункты **С**, смешанные **М**, большая таблица P-xx, отдельный модуль rules) **сохранён в git** в коммите перед этим файлом (`docs(plan): snapshot full plan …`).

## Объём MVP: только отчёт, без ОПИ

В этой версии **нет второго документа и нет сверки с ОПИ**: в форме **один путь** — к **отчёту** (TR/SR в `.docx`). Контекст для каждого вызова LLM — **только полный текст отчёта** (плюс промпт пункта). Так короче цепочка, меньше риск по лимиту контекста и проще демонстрация.

Сверка с **ОПИ** отложена — см. **[Задел на будущее](#задел-на-будущее-после-mvp)**.

## Неизменное окружение

- **Один путь** в форме: к **отчёту**; чтение `.docx` с диска после **windows_linux_path_map.txt**; файл **только read-only** (детали маппинга — в снимке полного плана в git, см. коммит `3c9c530`).
- **Выход пользователю — Markdown (`.md`):** для каждой зафиксированной **ошибки** в файл попадают **`id` пункта из YAML** и **подробное описание** (что не так, где в отчёте, рекомендация). Пункты **без ошибки** в итоговом MD либо не перечисляются, либо кратко суммируются одной строкой в начале (например «проверок: N, замечаний: K») — поведение зафиксировать в коде.
- **JSON** на сервере — полный лог шагов и сырые ответы модели, пользователю не отдаём.
- Документацию в продукте в этой итерации не расширяем; картинки — н/п.

## Формат итогового Markdown (минимальная договорённость)

Достаточно предсказуемой структуры, например:

```markdown
# Результаты проверки отчёта

- Проверок выполнено: …
- Найдено замечаний: …

## Замечания

### `id-из-yaml-1`
Подробное описание ошибки (многострочно при необходимости).

### `id-из-yaml-2`
…
```

Генерация — конкатенация/шаблон в коде **или** финальный ответ LLM, с последующей проверкой, что **каждый блок с ошибкой содержит валидный `id` из YAML**.

**Никакого Excel в MVP:** не трогаем `checklist_template.xlsx`, не заполняем колонки и листы.

## Единый принцип проверки (только LLM)

Для **каждого** пункта из YAML:

1. Берётся **готовый промпт** критерия.
2. В тот же запрос вкладывается **полный текст отчёта**. **ОПИ не подмешивается.**
3. Модель отвечает **структурированно** (JSON): прошёл ли пункт, если нет — **текст замечания**; поле **`id`** должно совпадать с пунктом YAML (дублировать в ответе для контроля парсера).
4. Следующий пункт — новый вызов с **новым промптом** и снова **полным** телом отчёта.

**Упрощение ценой токенов:** полный отчёт может повторяться в каждом вызове. Если контекст **не вмещает** — fallback (усечение с предупреждением в UI или подвызовы) — зафиксировать в реализации.

## Фронт

- Пользователь видит **какой пункт сейчас проверяется** (**`id`** и краткое название из YAML).
- Прогресс: N из M.
- По завершении — скачивание **`.md`**.

## Финальный шаг нейросети

После всех поштучных проверок — **финальный вызов LLM** на **сводку verdicts**. Задача: **согласовать список замечаний** (убрать дубликаты и противоречия, не ослабляя найденные ошибки), выдать структуру → из неё собирается **итоговый Markdown** (или финальная модель сразу возвращает MD при условии последующей валидации `id`).

## Стартовый набор пунктов (только «удобные для НС», внутри одного отчёта)

Исключаем строгую автоматику Word и всё, что требует ОПИ. Примеры смысловых проверок по одному тексту — как в предыдущей версии плана (титул внутри документа, цель, термины/ссылки внутри отчёта, согласованность разделов, заключение vs тело, формулы, единицы).

У каждого пункта в YAML — **свой устойчивый `id`** (например `titul-internal`, `goal-21`, или короткий код); **порядка 8–15** пунктов.

## Схема потока

```mermaid
flowchart LR
  UI[Путь к отчёту]
  Map[windows_linux_path_map.txt]
  Read[Текст отчёта]
  Loop[Для каждого пункта LLM]
  Final[Финальный LLM итог]
  MD[Markdown пользователю]
  Store[JSON на сервере]
  UI --> Map
  Map --> Read
  Read --> Loop
  Loop --> Final
  Final --> MD
  Loop --> Store
  Final --> Store
```

## Риски

- **Длина контекста** и **стоимость** — как раньше.
- **Финальный LLM** не должен удалять замечания без оснований — валидатор по числу/идентификаторам **`ошибка`**.
- **Markdown** проще Excel, но нет готовой укладки под ваш старый шаблон — перенос в таблицу вручную или в будущем экспорт в Excel.

## Задел на будущее (после MVP)

### Два документа: отчёт + ОПИ

- Второе поле пути, полный текст ОПИ в промптах, длинный контекст и стратегии укладывания в лимит — как в предыдущей редакции плана.

### Выгрузка в Excel по шаблону

- Заполнение [**checklist_template.xlsx**](/filer/users/rymax1e/MRO/report_checking/checklist_template.xlsx): колонка **E**, лист **`Детализация`**, привязка строк к **ID** шаблона (колонка **D**). Можно строить поверх того же внутреннего JSON, что и сейчас, добавив слой экспорта из Markdown/JSON в книгу.

### Прочее (из полного плана)

- Скриптовые проверки (**С**), смешанные **М**, модуль **rules**, нарезка по структуре Word — по мере зрелости продукта.

## Автотесты (только будущая версия MVP)

Цель — **не** покрывать текущий PDF-пайплайн, а закрепить контракты новой ветки: `.docx`, YAML, маппинг путей, job с прогрессом по пунктам, парсинг ответов LLM, итоговый Markdown и запись JSON на сервер. Все вызовы LLM в тестах — **фиксированные ответы / моки**, без сети и без реальной модели.

**Инфраструктура:** `pytest`, каталог `backend/tests/` (или зеркально структуре пакета), `conftest.py` с фикстурами (временные каталоги, сброс in-memory job store между тестами). При необходимости — `pytest.ini` с `pythonpath`, чтобы импортировать модули приложения.

### YAML чек-листа

- Успешная загрузка: несколько пунктов с **`id`**, **`prompt`**, опционально **`human_title`**; порядок сохраняется, если он важен для UI.
- Ошибки конфигурации: дубликаты **`id`**, отсутствие обязательных полей, пустой **`prompt`**, невалидный YAML — ожидаемое исключение или список ошибок валидации (как решите в коде).

### Извлечение текста из `.docx`

- Фикстура: маленький `.docx` с заранее известными абзацами/строками; проверка, что извлечённый плоский текст содержит ожидаемые фрагменты и порядок разумный (без проверки «read-only» на уровне ОС — достаточно того, что код не открывает файл на запись).
- Граничные случаи: пустой документ, документ только с пустыми абзацами — согласованное поведение (ошибка пользователю или явный флаг «нет текста»).

### Маппинг пути (`windows_linux_path_map.txt` + allowlist)

- Таблица кейсов: типичный Windows-путь → ожидаемый нормализованный путь под Linux после маппинга.
- Отклонения: путь вне белого списка корней, отсутствующий файл после резолва, неоднозначное правило маппинга — явная ошибка с понятным сообщением (для API/UI).

### Парсер ответа LLM по пункту (JSON)

- Валидные ответы: статусы вроде **`ok`**, **`error`**, **`na`**, **`unknown`** (или принятый в коде набор); при **`error`** — непустое описание для MD.
- **`id` в ответе не совпадает** с пунктом запроса — ошибка парсинга или жёсткий reject с логом в JSON.
- Невалидный JSON, обрезанный ответ, лишний текст вокруг JSON — устойчивое поведение (повтор не делаем в юнит-тесте, фиксируем один сценарий «как обрабатываем»).

### Оркестрация цикла по пунктам (без реального LLM)

- Подмена функции вызова модели: для N пунктов возвращаются заранее заданные JSON; проверка, что вызовов N, что в прогресс/статус job попадают **`id`** и заголовки из YAML.
- Сохранение **промежуточных verdicts** в JSON на диске: после шага файл содержит ожидаемую структуру (идемпотентность не обязательна, но форма полей — да).

### Финальный шаг (сводка)

- Мок финального ответа: на входе фиксированный список промежуточных вердиктов с **`error`**; на выходе структура не **теряет** ни одного такого **`id`** без явного правила слияния (инвариант из плана: не смягчать ошибки).
- Дубликаты/противоречия в моках — проверка, что постобработка или финальный парсер ведёт себя предсказуемо (как задокументируете).

### Сборка итогового Markdown

- По фиксированному списку нарушений: в тексте есть заголовок уровня 1, счётчики «проверок / замечаний» (если они в продукте), секция замечаний с подзаголовком на каждую ошибку в том же формате, что в примере MD выше (третий уровень + `id` пункта).
- **Валидация `id`:** каждый блок ошибки в MD соответствует **`id`** из загруженного YAML; лишние **`id`** в MD — падение теста.

### Job / HTTP API (новая версия)

- Старт проверки (путь к отчёту или уже зарезолвленный путь), **`GET` статуса**: поля «текущий пункт», «N из M», финальные состояния **`done`** / **`error`**.
- **`GET` результата `.md`** только в **`done`**; до готовности — 400; неверный `job_id` — 404.
- Убедиться, что **сырой JSON лог** не отдаётся эндпоинтом, предназначенным для пользователя (если отдельный internal route — он не в публичном контракте).

### Стратегия длины контекста (если реализована)

- При превышении лимита: заглушка или усечение с флагом в ответе job/UI — один тест с подменой лимита, без больших файлов.

### Фронтенд (по желанию, второй этап)

- Один-два теста с **`vitest`** + Testing Library: отправка формы с путём, отображение «сейчас проверяется …», появление ссылки/кнопки скачивания после успеха (с моком `fetch` к API).

## Порядок работ

1. YAML пунктов: **`id`**, **`prompt`**, опционально заголовок для UI.
2. Извлечение текста одного `.docx` read-only.
3. Backend: job, прогресс, цикл LLM, финальный вызов, сборка **`report_check.md`** (или аналогичное имя).
4. Фронт: путь к отчёту, «сейчас проверяется», скачивание **`.md`**.
5. Параллельно с пунктами 2–4 — добавлять тесты из раздела **«Автотесты»** по мере появления модулей.
