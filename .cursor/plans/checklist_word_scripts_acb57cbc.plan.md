# План (упрощённый): сначала только нейросеть, без скриптов

## Зачем упрощение

Нужно **показать применимость нейросети** в вашей работе по НК. Отдельно **скрипты не разрабатываем в этой фазе** — с ними и так всё понятно; время уходит на промпты, оркестрацию и выдачу результата в **привычный Excel**.

Полный вариант плана (скриптовые пункты **С**, смешанные **М**, большая таблица P-xx, отдельный модуль rules) **сохранён в git** в коммите перед этим файлом (`docs(plan): snapshot full plan …`).

## Неизменное окружение

- Два пути в форме: **ОПИ** и **отчёт**; чтение `.docx` с диска после **windows_linux_path_map.txt**; файлы **только read-only** (детали маппинга и Excel — в снимке полного плана в git, см. коммит `3c9c530`).
- Выход пользователю: заполненный [**checklist_template.xlsx**](/filer/users/rymax1e/MRO/report_checking/checklist_template.xlsx) — колонка **E** «Результат проверки», лист **`Детализация`** для подробностей при **`ошибка`** (как в полном плане).
- **JSON** на сервере — лог промежуточных и финальных шагов, пользователю не отдаём.
- Документацию в продукте в этой итерации не расширяем; картинки — н/п.

## Единый принцип проверки (только LLM)

Для **каждого** пункта из короткого списка (ниже):

1. Берётся **готовый промпт** критерия (из YAML/конфига).
2. В тот же запрос вкладывается **полный текст отчёта** (извлечённый из `.docx`). Если пункт требует сверки с ОПИ — в запрос добавляется и **полный текст ОПИ** (или сначала отчёт + ОПИ в фиксированном порядке в одном сообщении — зафиксировать в шаблоне промпта).
3. Модель отвечает **только на этот пункт**; ответ — **структурированный** (например JSON): значение для **E** (`не проверено` / `проверено` / `ошибка` / `не применимо`), краткий комментарий, при `ошибка` — текст для **`Детализация`**.
4. Переход к **следующему** пункту — новый вызов с **новым промптом** и снова **полным** телом отчёта (и ОПИ при необходимости).

**Упрощение ценой токенов:** один и тот же длинный отчёт может передаваться в каждом вызове — это сознательно, чтобы не писать нарезку и скрипты выборки фрагментов на первом этапе. Если контекст модели **не вмещает** документ — в реализации заранее предусмотреть fallback (укороченный текст с предупреждением в UI, или разбиение с дисциплиной «один пункт — несколько подвызовов»); это **единственное** отклонение от «всегда целиком».

## Фронт

- Пользователь видит **какой пункт сейчас проверяется** (id из колонки **D**, короткое имя из шаблона / YAML).
- Прогресс: N из M.
- По завершении — скачивание **.xlsx**.

## Финальный шаг нейросети

После всех поштучных проверок выполняется **финальный вызов LLM**: на вход подаётся **сводка всех промежуточных verdicts** (таблица/JSON). Задача — **согласовать итоговый чеклист**: выровнять формулировки, убрать противоречия, при необходимости переформулировать комментарии для колонки **E** и листа **`Детализация`**, не меняя смысл без оснований (в промпте явно: «не ослаблять найденные ошибки»).

Затем заполнение Excel из **финальной** структуры (после парсинга ответа).

## Стартовый набор пунктов (только «удобные для НС»)

Исключаем всё, что опирается на **точный** поиск полей Word, подсчёт страниц, строгие регулярки имён файлов без LLM — их вернём позже скриптами. В MVP оставляем **смысловые** проверки по полному тексту, например:

- Согласованность **титула / шапки** отчёта с **ОПИ** (реквизиты, название, ревизия).
- **Применяемость** (п.1) vs формулировки в ОПИ.
- **Цель** (п.2.1) — полнота и соответствие ожидаемой роли раздела.
- **Ссылочные документы / термины** — на уровне «нет явных противоречий и дыр в списке относительно упоминаний в тексте» (без строгой нумерации скриптом).
- **Согласованность ключевых глав 3.x** с описанием работ/модификации из ОПИ (текстом).
- **Общая оценка / заключение** — не противоречат ли телу отчёта.
- **Формулы** — осмысленность и согласованность с окружающим текстом (как в полном плане).
- **Единицы ГОСТ 8.417** — грубые нарушения в тексте (как эвристика глазами модели).

Точный перечень и **template_row_id** для каждой строки — в YAML; пунктов немного (**порядка 8–15**), чтобы уложиться в демонстрацию.

## Схема потока

```mermaid
flowchart LR
  UI[Два путей]
  Map[windows_linux_path_map.txt]
  Read[Текст ОПИ и отчёта]
  Loop[Для каждого пункта LLM]
  Final[Финальный LLM итог]
  XLSX[Excel из шаблона]
  Store[JSON на сервере]
  UI --> Map
  Map --> Read
  Read --> Loop
  Loop --> Final
  Final --> XLSX
  Loop --> Store
  Final --> Store
```

## Риски

- **Длина контекста** — полный отчёт повторяется в каждом вызове; нужен лимит модели и честное предупреждение в UI при усечении.
- **Стоимость и время** — линейно от числа пунктов × размер документа; для демо согласовать небольшой набор пунктов.
- **Финальный LLM** может переформулировать — жёстко проверять парсером, что статусы **`ошибка`** не пропали без обоснования.

## Порядок работ

1. YAML пунктов + промпты + привязка к **D** в шаблоне.
2. Извлечение текста `.docx` read-only.
3. Backend job с прогрессом, цикл LLM по пунктам, финальный вызов, сборка **.xlsx**.
4. Фронт: пути, «сейчас проверяется», скачивание файла.
