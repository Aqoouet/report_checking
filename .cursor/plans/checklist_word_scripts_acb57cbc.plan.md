# План: Word + ОПИ по путям, чеклист для людей в Excel

## Зачем меняем подход

Раньше использовали **PDF**, потому что в него просто вставлять заметки. Сейчас цель — **проверка по регламенту** и **удобный чеклист**, исходники у вас в **Word**.

**Файлы через браузер передавать нельзя** (ограничение ИТ). В интерфейсе — **два поля ввода** с полными путями к **ОПИ** и к **отчёту**. На сервер уходят строки путей; содержимое файлов по HTTP не грузится. Пути переводятся в Linux по **файлу соответствий** (см. ниже), затем сервис **читает .docx с диска**.

**Защита исходников:** оба входных документа (**ОПИ** и **отчёт**) на всём пайплайне обрабатываются **только для чтения**: открытие файлов и разбор `.docx` (архив ZIP) без режима записи, **никакого сохранения** изменений в эти пути, вспомогательные временные файлы — только при необходимости в отдельном каталоге/буфере, не подменяя оригиналы. Цель — исключить порчу файлов скриптом или библиотекой при сбое.

**Документацию (README и т.п.) в рамках этой итерации сознательно не закладываем** — чтобы сэкономить время; при необходимости добавите позже.

## Что получает пользователь на выходе

- **Excel (.xlsx)** — не «с нуля», а **заполненный шаблон** [`checklist_template.xlsx`](/filer/users/rymax1e/MRO/report_checking/checklist_template.xlsx): копия в рабочий файл, разметка и листы-справочники сохраняются.
- **JSON** копится **на сервере** для статистики и отладки, пользователю не показываем.

## Шаблон Excel: как заполняем

**Исходник в репозитории:** [`checklist_template.xlsx`](/filer/users/rymax1e/MRO/report_checking/checklist_template.xlsx).

**Основной лист чеклиста** (в текущем файле — **`MRO_А_23.06.2023`**; имя может меняться с датой версии шаблона): при реализации задать **имя листа в конфиге** или находить лист по признаку шапки (в строке заголовка блока есть **`Результат проверки`** в **колонке E**, колонка **D** — **`ID`** числовой идентификатор пункта, **B** — текст требования). **Только колонка E** в строках пунктов (под шапкой) заполняется итогом проверки; остальные колонки шаблона **не перезаписывать** без необходимости (веса, формулы).

**Допустимые значения в E** — в терминах шаблона (согласованы с листом **`Inputs`**, колонки **B/D**): русские подписи **`не проверено`**, **`проверено`**, **`ошибка`**, **`не применимо`** (английские эквиваленты в коде маппить на эти четыре). Автоматика должна писать **ровно те строки**, что ожидает шаблон (в т.ч. для выпадающих списков/валидации, если они заданы в книге).

**Лист `Inputs`** — справочник (легенда статусов, типы документов и т.д.): **не изменять** при генерации; только читать при необходимости.

**Отдельный лист с подробностями при ошибке:** в шаблоне сейчас **нет** выделенного листа под длинный текст замечаний — **добавить** в книгу (в шаблоне или программно при первом сохранении) лист, например **`Детализация`** (имя зафиксировать в коде/конфиге), структура минимум:

- идентификатор пункта (**`ID`** как в колонке D основного листа и/или **P-xx** из внутренней схемы);
- краткая метка пункта (опционально, дублировать начало текста из колонки B для удобства чтения);
- **`Подробное описание`** — полный текст замечания, цитата, пояснение модели, шаги исправления.

**Правило заполнения:** строка на листе **`Детализация`** создаётся **для каждого пункта с результатом `ошибка`** (при необходимости — и для `не применимо` с пояснением, если в продукте нужна трактовка); для `проверено` и `не проверено` строки детализации **не добавлять** (или оставлять пусто — выбрать одно поведение и держать его в коде).

**Связка YAML ↔ Excel:** в YAML на каждый автоматизируемый пункт хранить **`template_row_id`** (число из колонки **D**) или однозначную ссылку на строку основного листа, чтобы оркестратор знал, **в какую строку писать E** и какую **ID** проставить в **`Детализация`**.

Генерация книги: **openpyxl** (или аналог) — открыть копию шаблона, записать значения, сохранить в поток для отдачи пользователю.

## Файл соответствия путей Windows → Linux

- **Рабочий файл** — по умолчанию **`windows_linux_path_map.txt` в корне репозитория** ([`/filer/users/rymax1e/MRO/report_checking/windows_linux_path_map.txt`](/filer/users/rymax1e/MRO/report_checking/windows_linux_path_map.txt)): соответствие префиксов путей Windows → Linux для доступа к **ОПИ и отчётам (.docx)** на сервере. При реализации путь можно переопределить через переменную окружения **`WINDOWS_LINUX_PATH_MAP_FILE`**, если корень проекта на сервере другой.
- **Формат строк** — тот же, что в эталоне **`/filer/wps/scripts/Apps/Geometry_search/scan_paths.txt`** (можно переносить/копировать строки оттуда; парсер в коде повторяет этот формат).
- Строки вы **заносите вручную** (одна строка — одно правило замены).
- После замены: проверка что файл существует, расширение `.docx`, путь остаётся в **разрешённых корнях**; далее открытие **только для чтения** (см. выше).

## ОПИ

- В проверке всегда участвуют **ОПИ** и **отчёт** (два пути из формы).

## Проверка формул

В 005С для НК подстановку в формулы не требуют; вы **делаете это в продукте**. Чтобы не застрять на краевых случаях (единицы, запись дробей, нестандартные обозначения), **всю проверку формул в первой версии отдаём нейросети**: в промпт уходит фрагмент с формулой и окружающий текст (и при необходимости кусок таблицы), модель оценивает согласованность подстановки и итогового числа. Отдельный «калькулятор формул» в коде для первой версии **не обязателен**.

## Работа с нейросетью (как сделать разумно)

Цель — **меньше токенов, предсказуемый парсинг, меньше галлюцинаций**.

1. **Максимум детерминизма в коде:** пункты **С** не трогают LLM; для **М** в модель уходит уже **сжатый факт** из скрипта (извлечённые строки, списки, флаги), а не весь документ — LLM решает только «смысловой» слой.
2. **Нарезка контекста:** большие главы и **P-22** — **отдельные вызовы по разделам/окнам** с указанием в промпте «это фрагмент отчёта, раздел N»; не отправлять весь отчёт одним сообщением.
3. **Структурированный ответ:** просить выход в **строгом JSON** (или XML) с полями `status`, `comment`, при необходимости `evidence_quote`; в коде — парсинг и валидация схемы, при ошибке — **один повтор** с укороченным промптом или fallback «нужна ручная проверка».
4. **Пакетирование:** где логика однотипна (несколько коротких вопросов по одному куску текста), **объединять в один вызов** с массивом подзадач — меньше round-trip, но не раздувать промпт сверх лимита модели.
5. **Параметры:** для факт-чеков — **низкая температура**; системный промпт фиксировать «не выдумывать цитаты; если данных нет — явно сказать».
6. **ОПИ + отчёт:** в каждый релевантный запрос класть **только нужные выдержки** из ОПИ (титул, применяемость и т.д.), а не полный ОПИ целиком, если он большой.
7. **Учёт стоимости и задержки:** оркестратор ведёт **очередь LLM-задач** с лимитом параллелизма (например 2–4), чтобы не упираться в rate limit и не дублировать один и тот же огромный контекст в пяти параллельных запросах без необходимости.

*(Детали модели и провайдера — по вашей инфраструктуре; принципы выше не зависят от конкретного API.)*

## Порядок работ (по шагам)

1. Подключить **`windows_linux_path_map.txt`** и **только чтение** двух `.docx` с диска по преобразованным путям.
2. YAML (или аналог) с пунктами **P-xx** и полем метода С/Н/М/—.
3. Реализовать **С**-пункты из таблицы выше; API + фронт с двумя путями; выгрузка **по `checklist_template.xlsx`** (колонка **E**, лист **`Детализация`**) + внутренний **JSON**; чтение `.docx` только read-only.
4. Промпты **Н** и **М** с участием ОПИ; **формулы только через Н** (P-50).
5. Убрать PDF-сценарий для пользователя.
6. Картинки — не делать.

## Риски

- Неверная строка в **`windows_linux_path_map.txt`** (в корне проекта) → целевой `.docx` не найден; понятные ошибки в UI.
- Нейросеть по формулам может ошибаться — в чеклисте допускать формулировку «рекомендация модели, нужна ручная сверка».
- Большие отчёты — резать контекст по разделам, не слать целиком (см. раздел про нейросеть).
- Смена **имени основного листа** в новой версии `checklist_template.xlsx` (сейчас с датой в названии) — держать **конфиг или авто-поиск** шапки, иначе запись уйдёт не в тот лист.
- Лист **`Детализация`** должен быть согласован с процессом НК (одна строка на ошибку, без дублей **ID**).
